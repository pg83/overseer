## CWD vs WORKSPACE

Your current working directory is a session-storage directory the harness uses to remember context across runs — it is NOT the project tree. Do NOT run git, read source, or write files there.

Instead use the absolute path passed in your input (typically `WORKSPACE`, sometimes a role-specific name like `MERGER_WORKTREE`). Either `cd "$WORKSPACE"` first or pass paths explicitly to every command (`git -C "$WORKSPACE" ...`, absolute file paths to file tools).

## Output protocol — typed JSON events

**Emit structured signals by writing JSON lines directly in your text response.** No tool, no function, no helper — type the JSON as part of your reply.

When the prompt says "emit a verdict" or "emit a task event", write a line like

    {"type": "verdict", "verdict": "READY", "detail": "..."}

as one of the lines in your reply. Lines whose first character is `{` are parsed as events; everything else is private reasoning prose.

Rules:

- One event per line. Put each event on its own line, no markdown wrappers.
- Keep each event on a single line — escape newlines inside string fields as `\n`.
- Stick to the events documented for your role.
- Interleave prose and events in any order — only events have semantic effect.

Common events available to every role:

    {"type": "message", "text": "<one-line note posted to the team chat>"}
    {"type": "replan", "reason": "<why the task DB needs adjustment>"}

`message` events post to the shared team chat (`MESSAGES_LOG`) where every role across every ticket reads each other. Use them for status, surprises, ambiguities, and decisions worth flagging — not routine tool progress, not the final verdict, not sensitive data. Split multi-line content into multiple `message` events or summarize it; escape literal newlines as `\n`.

`replan` events nudge the replanner — emit when you notice the task DB itself needs adjustment (missing prerequisite, wrong split, etc.).

`MESSAGES_LOG` (path in your input header) is the team's shared chat — every `message` from every role across every ticket, append-only `<ts>\t<role>\tT-<n>\t<text>`. Read it (or grep for your ticket / a teammate's role) for context on what others noticed; write to it by emitting `message` events.
