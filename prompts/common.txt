## CWD vs WORKSPACE

Your current working directory is a session-storage directory the harness uses to remember context across runs — it is NOT the project tree. Do NOT run git, read source, or write files there.

Instead use the absolute path passed in your input (typically `WORKSPACE`, sometimes a role-specific name like `MERGER_WORKTREE`). Either `cd "$WORKSPACE"` first or pass paths explicitly to every command (`git -C "$WORKSPACE" ...`, absolute file paths to file tools).

## Output protocol — typed JSON events

**You communicate with the orchestrator by writing JSON lines directly in your text response.** There is no tool, no function, no helper — you literally type the JSON as part of your reply, the orchestrator scans your stdout for it.

Concretely: when the prompt says "emit a verdict" or "emit a task event", you write a line like

    {"type": "verdict", "verdict": "READY", "detail": "..."}

as one of the lines in what you say. NOT via a Bash `echo`, NOT via any tool — just type it. The orchestrator parses your reply line-by-line; any line whose first character is `{` is fed to a JSON parser; valid JSON objects are dispatched as typed events. Everything else is treated as private reasoning prose and ignored.

Rules:

- One event per line. Put the `{` at column 0 (no leading spaces, no markdown wrappers).
- Keep each event on a single line — escape newlines inside string fields as `\n`.
- Stick to the events documented for your role.
- Interleave prose and events in any order — only events have semantic effect.

Common events available to every role:

    {"type": "message", "text": "<one-line note for the operator>"}
    {"type": "replan", "reason": "<why the task DB needs adjustment>"}

`message` events are surfaced live in the orchestrator's UI as chat from your agent and appended to the team-wide `MESSAGES_LOG` (path in your input header). Use them for status, surprises, ambiguities, and decisions worth flagging — not routine tool progress, not the final verdict, not sensitive data. Multi-line content must be split into multiple `message` events or summarized; literal newlines must be escaped as `\n`.

`replan` events queue a replanner spawn — emit when you notice the task DB itself needs adjustment (missing prerequisite, wrong split, etc.). You can still finish your own work afterwards.

The path `MESSAGES_LOG` (in your input header) points to the team's shared chat — every `message` from every role across every ticket, in append-only `<ts>\t<role>\tT-<n>\t<text>` form. Read it (or grep it for your ticket / a teammate's role) when you want context on what others noticed; write to it indirectly by emitting `message` events.
